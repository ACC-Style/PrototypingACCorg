Canonical patterns (backend recipes)

Three tab kinds. Backend should pick one recipe per nav page based on SubPagesExist and child count.

A. Plain link (no children)

  • <li id="nav_tab_*" class="relative flex_auto items_center m_0 p_0 flex flex_row"> — no nav-has-submenu
  • One <a class="nav-link-arches flex_100 flex_shrink nav-submenu-label-arches nowrap p-x_3 p-y_3 a:bg_accent-n3 h:bg_accent-n1 c_white not-link block h:undecorated">
  • Current page: add active on <li> + aria-current="page" on the link

B. Dropdown — 1 column (1–5 children)

  • <li id="nav_tab_*" class="relative flex_auto nav-has-submenu items_center m_0 p_0 flex flex_row">
  • Toggle <button class="nav-link-arches flex_none nav-submenu-toggle … display_none:md block"> with interestfor + commandfor + command="toggle-popover"
  • Label <a class="nav-link-arches flex_100 flex_shrink nav-submenu-label-arches … block h:undecorated" interestfor="…">
  • Panel <div id="nav_menu_*" popover="auto" anchor="nav_tab_*" class="nav-submenu-popover dropdown-menu-arches font_0 m-x_3 shadow_overlap-light br_black-1 br_solid br_1 br-bl_round br-br_round font_copy br-t_0 br_none br_solid:md br_square z_5">
  • Inner grid: grid columns_1 p-t_2 p-b_3 p-x_3:md — no gap-x_3 (single column)
  • One .dropdown_column → ul.ul_none lh_2 m_0 flex flex_column gap_1 c_black text_left font_ui
  • Child <li> has no empty class
  • Each child link: h:bg_accent-4 a:bg_accent-3 block p-y_3 p-x_4 br_radius not-link c_inherit-all wrap sub-menu-item-arches h:undecorated wrapping span.text-indent_n1.block.m-l_4

C. Dropdown — 2 columns (> 5 children) — additive on B

  • Why: long child lists need a wider panel and a visual split so items stay scannable
  • Same as B, plus these deltas only:
  • Panel adds w_auto:md
  • Inner grid adds columns_2:md gap-x_3 (gap-x_3 is C-only — column gutter)
  • Split children into two .dropdown_column blocks (≈ half each)
  • 2nd column only: br-l_1 br_solid:md br_none br_black-2 p-l_3:md

Shared behavior (all dropdown tabs)

  • commandfor / command="toggle-popover" → click open (mobile + desktop)
  • interestfor on toggle + label → hover open (desktop)
  • anchor="nav_tab_*" on popover → panel tethers to the tab <li>
  • Scoped CSS anchor-name: --nav-tab + anchor-scope — no per-tab CSS
  • Breakpoint family: md — min-width: 768px / max-width: 767.98px
  • Mobile in-flow (open): Popover API always promotes open panels to the top layer, so they cannot push following nav items in document flow. That needs a separate mobile pattern (not stripping popover at runtime — that broke desktop). TBD with care.

Resolved decisions

  1. Home: removed nav-has-submenu (plain = recipe A)
  2. Guidelines: added id="nav_tab_guidelines"
  3. All top-level labels: h:undecorated
  4. All panels: dropdown-menu-arches font_0 (no Bootstrap dropdown-menu)
  5. gap-x_3 is additive for recipe C only
  6. Empty class="" stripped from submenu <li>s
  7. Page CSS aligned to md / 768 (see breakpoint review below)

Breakpoint review (item 7)

Markup utilities use :md (768). Scoped CSS must use the same cut so toggler hide, row layout, and popover positioning flip together.

  • Mobile main-nav panel — was max-width: 991.98px → now max-width: 767.98px. Anchors the collapsed menu under the bar only while the Menu button is shown.
  • Desktop: hide toggler / force nav visible — already min-width: 768px. Keep.
  • Desktop: tab anchor + fixed submenu — already min-width: 768px. Keep.
  • Desktop hover / interest — already min-width: 768px + hover: hover. Keep.
  • Mobile: submenu positioning — below md, attempt position: static on open panels. Note: Popover top-layer still overlays rather than expanding the nav; a safe in-flow mobile recipe is still TBD.
